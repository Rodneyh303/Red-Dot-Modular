## What it shows (spec)

A 16-lane bar view over the 16-step grid â€” one narrow lane per voice (row N = voice N),
like the Sands lane editors but display-only and narrower.

- **Row = voice**, top to bottom (row 1 = mono/V1, rows 2..16 = poly voices). Row number IS
  the voice number â€” no per-voice colour needed for identity.
- **Left gutter label:** note name + octave (e.g. `A3`) for each active voice; `â€“` inactive.
- **Bars over the 16-step grid:** each note drawn as a horizontal bar from its start step to
  its end. Bar end lands at **sub-step resolution** (mid-step for triplet / 1/32), read from
  the note-length index â€” so you see the true note-off, not a step-quantised approximation.
- **Colour = NOTE TYPE, not voice:** single note / tie / legato / accent (+ inactive grey).
  Voice identity is the row, so colour is free to carry note semantics. Accent = brightest /
  brand-red-related (emphasis = the dot.modular signal colour).
- **Tie components:** vertical cross-lines within a held span marking the tied sub-notes.
- **Held-over caret:** a mark on the LEFT of step 1 when a note continues from the previous
  bar; (optional) a right-edge caret when a note continues out past step 16.
- **View modes:** Notes / Velocity / Prob (all read-only). Zoom (Ã—1/Ã—2/Ã—4 time scale) +
  Follow (keep current phase in view). Current phase position = a red vertical line.

## Data availability â€” the read-only "pure observer" check (RESULT: GO, with one note)

Lantern reads existing engine state; it adds NO engine fields. Confirmed available:

| Needs | Source (read-only) |
|---|---|
| Per-voice pitch (â†’ note name) | `GateState.currentPitchV` (mono `engine.gs`; poly `engine.voices[i].gs`) |
| Gate on/off | `GateState.gateHeld` |
| Note length (incl. sub-step) | `nvIdx` + `gs_noteSteps(nvIdx)` / `noteValueSteps` |
| Held-remaining (cross-bar) | `GateState.holdRemain` |
| Note TYPE (single/tie/legato/rest/new) | `StepResult.decision` (`MonoDecision` enum) â€” already computed |
| Accent | `StepResult.accented` (mono) / `voices[i].accented` (poly) |
| Active voice count | `engine.numPolyVoices` |

**The one architectural note:** `StepResult`/`MonoDecision` is only the **most recent** step
(`engine.lastStepResult`), not a 16-step history. To draw a whole bar of note-types, Lantern
keeps its **OWN per-voice 16-step display ring buffer**, sampled from the engine each step
edge. This is a legitimate scope/logic-analyser pattern â€” the history lives in the WIDGET,
not the engine, so the observer stays pure (reads engine, writes only its own display state).

Small self-contained helper needed (no engine change): `1V/oct â†’ semitone â†’ "A3"` for the
labels. Lives in the Lantern widget (or a tiny `NoteName.hpp`).

**Cross-bar ties** (the hard part): `holdRemain` gives "still ringing + how long", so the
window-boundary carets are computed from it; the multi-bar tie cross-lines come from the
accumulated history of successive `Tie` decisions in Lantern's own buffer. Solved in render,
no engine state added.

## Piano-roll alternative (rejected)

A piano-roll (pitch axis) view was considered and dropped: it needs vertical space 16 voices
don't allow, obscures the per-voice/per-step structure that's useful for debugging, and is a
much larger build. The bar view reuses existing lane geometry and answers "what is each voice
doing at each step" directly â€” the right question for a debug tool.
